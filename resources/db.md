# NEXUS - PostgresQL

## 1. Convenções de Escrita e Padrões

Para manter a consistência e evitar conflitos entre diferentes bancos de dados (como PostgreSQL e MySQL), as seguintes convenções foram adotadas:

* **Snake Case:** Todos os nomes de tabelas e colunas utilizam letras minúsculas separadas por underscore (ex: `user_mission_attempt`).
* **Aspas em Nomes Reservados:** Palavras que podem ser reservadas pelo SQL (como `"user"`, `"level"`, `"number"`, `"order"`) devem ser sempre escritas entre aspas duplas.
* **Identificadores:** Chaves primárias são sempre do tipo `bigint` e utilizam `GENERATED BY DEFAULT AS IDENTITY` para autoincremento.
* **Integridade:** Uso extensivo de `ON DELETE CASCADE` para garantir que, ao remover um usuário ou missão, os registros históricos dependentes não fiquem "órfãos".
* **Prefixos de Constraints:**
* `pk_`: Primary Key
* `fk_`: Foreign Key
* `uk_`: Unique Key
* `ck_`: Check Constraint

## 2. Entidades e Relacionamentos

### 2.1 Núcleo de Usuário e Progressão

Responsável por armazenar quem são os jogadores e em que estágio de evolução eles se encontram.

* **`user`**: Cadastro básico. O email é o identificador único de negócio.
* **`level`**: Tabela de referência que define a régua de XP necessária para avançar.
* **`user_stat`**: O "perfil de progresso". É uma extensão da tabela `user` (Relação 1:1), guardando o XP atual e a ofensiva (*streak*).

### 2.2 Conteúdo Educacional (Missões)

Define a estrutura do jogo: o que o usuário precisa aprender e quem ele deve enfrentar.

* **`mission`**: A unidade principal de conteúdo. Pode ter uma missão pré-requisito (auto-relacionamento).
* **`boss`**: Um desafio final opcional para a missão (Relação 1:1 com `mission`).
* **`question`**: Perguntas associadas a uma missão. A coluna `"order"` garante a sequência pedagógica.
* **`alternative`**: Opções de resposta para cada pergunta.

### 2.3 Execução e Histórico

Registra a interação em tempo real do usuário com o conteúdo.

* **`user_mission`**: Tabela associativa que vincula um usuário a uma missão e guarda o seu `status` (IN_PROGRESS/COMPLETED) e o melhor resultado obtido.
* **`user_mission_attempt`**: Cada vez que um usuário inicia uma missão, uma nova tentativa é gerada. Isso permite auditoria de tempo (`start_at`, `end_at`) e desempenho.
* **`user_response`**: O detalhamento de cada resposta dada em uma tentativa específica.

### 2.4 Recompensas

O sistema de gratificação por objetivos alcançados.

* **`achievement`**: Definição dos troféus. Estão geralmente vinculados à conclusão de uma missão específica.
* **`user_achievement`**: O "mural de troféus" do usuário. Registra o momento exato da conquista.

## 3. Matriz de Cardinalidade

| Relação | Tipo | Descrição |
| --- | --- | --- |
| `user` → `user_stat` | **1:1** | Todo usuário tem exatamente um registro de estatísticas. |
| `mission` → `question` | **1:N** | Uma missão pode ter várias perguntas. |
| `question` → `alternative` | **1:N** | Uma pergunta tem várias opções, mas apenas uma costuma ser a correta. |
| `user` → `mission` | **N:N** | Muitos usuários fazem muitas missões (resolvido por `user_mission`). |
| `attempt` → `user_response` | **1:N** | Uma tentativa de missão gera várias respostas individuais. |

## 4. Regras de Negócio Implementadas (Constraints)

1. **XP Positivo:** `xp_current` e `xp_required` nunca podem ser negativos.
2. **Unicidade de Missão:** Um usuário não pode ter dois registros de progresso para a mesma missão (`uk_user_mission`).
3. **Ordem de Questões:** Não podem existir duas perguntas com o mesmo número de ordem dentro de uma mesma missão (`uk_question_mission_order`).
4. **Status Controlado:** O campo `status` em `user_mission` aceita apenas os valores fixos `IN_PROGRESS` ou `COMPLETED`.
